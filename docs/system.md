# x-skills 设计规格书

## 系统解决的核心问题

**AI 辅助开发时，如何保证过程质量和知识不丢失。**

| 子问题 | 表现 | 对应 skill |
|--------|------|-----------|
| **规范执行一致性** | 每个 session 都按同一套规则操作，不靠记忆 | xLog、xReview、xCommit |
| **工作流连贯性** | 调试→补日志→修复→验证→提交，不断链 | xDebug→xLog→xTest→xCommit |
| **过程记录留存** | 调试记录、决策、测试问题不随 session 消失 | DEBUG-LOG、DECIDE-LOG、TEST-ISSUES |

## 系统设计原则

### 通用性

- **不硬编码项目细节**：路径、命令、目录结构、框架名称一律不写死在 skill 中
- **差异通过探测抽象**：标志文件探测 → SKILL-STATE.md 存储 → 各 skill 读取；项目特定信息只在 SKILL-STATE.md 中出现一次

### 单一职责

- **一个 skill 解决一类问题**：日志是 xLog 的事、调试是 xDebug 的事，职责不交叉
- **边界通过决策流定义**：每个 skill 的能力范围 = 它的决策流覆盖范围，决策流之外的事不做
- **skill 间不耦合内部逻辑**：只通过核心文件和衔接协议交互，不直接读写对方的中间状态

### 持久化

- **过程产出写入核心文件**：调试记录、决策、测试问题、规范——凡是跨 session 有价值的信息都持久化到文件，不只在对话中存在
- **核心文件是 skill 间的数据合同**：语义稳定；读写双方遵循同一格式，修改格式需同步更新所有消费方

### 用户控制

- **选项优先于打字**：能用 AskUserQuestion 选项就不让用户打字，Other 兜底自由输入
- **先展示后执行**：有副作用的操作（修改代码、写入文件、提交）先展示方案，用户确认后执行
- **每轮只问一个问题**：不堆叠多个决策点，降低用户认知负担

### 初始化

- **探测优先**：能从项目自动识别的信息不让用户手动配置
- **幂等安全**：重复运行初始化不破坏已有配置和用户自定义内容
- **增量补全**：不强制一次性初始化所有 skill，`ensure` 模式只处理未初始化的部分

### 衔接

- **显式决策点入口**：skill 间衔接通过目标 skill 的决策点入口调用，不绕过决策流直接执行内部逻辑
- **衔接调用 vs 提醒**：有直接因果关系（测试失败→调试）用衔接调用，自动传递上下文；仅信息关联（提交时可能遗漏文档）用提醒，不自动跳转
- **数据通过核心文件传递**：衔接时传递的是核心文件中的条目编号或路径，不传递对话中的临时状态
- **用户触发 vs agent 触发**：跨 skill 衔接由用户选择触发（如"立即修复""提交变更"）；skill 内部调用子 agent（如 xDebug 调用 xLog targeted）可自动触发

## 术语表

| 术语 | 定义 |
|------|------|
| 项目文件 | 生成在用户项目中、由 skill 运行时创建和维护的文件。分为规范（\*-RULES.md）、记录（\*-LOG.md / \*-CHECKLIST.md / \*-ISSUES.md）、状态枢纽（SKILL-STATE.md）、工具（run.sh） |
| 参考文件 | `references/` 下的设计时产出，承载 guideline / standard / template / protocol / artifacts |
| 决策流 | skill 的 D1→D2→... 决策点序列，定义 skill 的能力边界 |
| 决策点 | 决策流中的一个节点（Dn），对应一次用户选择或 agent 判断 |
| 阶段 0 | xBase 初始化时为某个 skill 创建项目文件的过程（见下方说明） |
| 衔接调用 | skill 间的跳转，通过目标 skill 的决策点入口调用，自动传递核心文件中的条目编号或路径 |

---

## 阶段 0

每个 skill 的决策流（D1→D2→...）描述的是 skill 自身的运行时逻辑。但运行前有一个前提：项目文件必须已存在。阶段 0 就是这个前提的保障——由 xBase 统一执行，为每个 skill 创建所需的项目文件。

- **规范类**（\*-RULES.md）：agent 在 guideline 指导下，从项目探测数据生成
- **记录类**（\*-LOG.md / \*-CHECKLIST.md / \*-ISSUES.md）：按 template 创建空文件结构

阶段 0 对每个项目文件做四态判定：

| 状态 | 含义 | 处理 |
|------|------|------|
| 需创建 | 文件不存在 | 按 guideline 或 template 生成 |
| 迁移候选 | 文件存在但结构不匹配当前格式定义 | 提示用户选择迁移或保留 |
| 需更新 | 文件存在且结构匹配，但内容过时 | 合并更新，保留用户自定义内容 |
| 已就绪 | 文件存在且内容与探测结果一致 | 跳过 |

## 文件体系

### x-skill 内部文件

存放在 `.claude/skills/x{name}/` 下，随 x-skills 仓库版本管理。不含项目特定信息，可跨项目复用。

#### SKILL.md（流程编排）

- **是什么**：skill 的决策流程和执行指令
- **怎么来的**：设计时编写
- **包含什么**：参数处理、决策点、阶段流程、衔接规则
- **给谁用**：Claude 加载后按流程执行
- **什么时候用**：skill 被触发时

#### 参考文件（`references/` 目录）

设计时产出。SKILL.md 负责流程编排，参考文件承载指南、标准、模板和协议。agent 按需读取，不预加载。

**Guideline（生成指南）** — 命名：`*-guideline.md`

- **是什么**：规范类项目文件的生成指南（创造性过程）
- **怎么来的**：AI 联网搜索该领域的行业最佳实践，分析总结凝练而成
- **包含什么**：规范应覆盖的维度、每个维度的质量要求、反模式、结构模板
- **给谁用**：agent 生成规范类项目文件时，指导如何从项目数据产出高质量规范
- **什么时候用**：生成或评估规范类项目文件时
- **不包含什么**：项目特定信息、具体规则条目

**Standard（书写标准）** — 命名：`*-standard.md`

- **是什么**：记录类项目文件的书写标准（机械性过程）
- **怎么来的**：设计时定义
- **包含什么**：字段质量标准、编号规则、分类规则、状态流转规则
- **给谁用**：agent 向记录类项目文件追加条目时，指导怎么写
- **什么时候用**：写入记录时对照
- **不包含什么**：结构骨架（template）、项目特定数据

**Template（模板）** — 命名：`*-template.md`

- **是什么**：记录类项目文件的结构骨架
- **怎么来的**：设计时预定义
- **包含什么**：段落标题 + 占位符，无说明文字
- **给谁用**：agent 初始化记录类项目文件时，提供空文件结构
- **什么时候用**：创建或重建记录类项目文件时
- **不包含什么**：字段说明、质量要求、项目特定数据

**Protocol（协议）** — 命名：`protocol-*.md`

- **是什么**：skill 执行流程中的步骤规程
- **怎么来的**：设计时定义
- **包含什么**：具体的操作步骤、判断条件、前置检查
- **给谁用**：agent 执行 SKILL.md 中引用该协议的步骤时
- **什么时候用**：SKILL.md 流程中按需读取
- **不包含什么**：项目文件内容、项目特定数据

**Artifacts（制品清单）** — 命名：`artifacts.md`

- **是什么**：该 skill 产出的所有项目文件清单
- **怎么来的**：设计时定义
- **包含什么**：项目文件名称、类型、创建时机
- **给谁用**：xBase 初始化时，知道该为这个 skill 创建哪些文件
- **什么时候用**：初始化时
- **不包含什么**：项目文件的内容或格式

#### 脚本文件（`scripts/` 目录）

xBase 提供的 .py 辅助工具，供 SKILL.md 流程中调用。

**state.py** — 位置：`xbase/scripts/`

- **是什么**：SKILL-STATE.md 的读写工具
- **包含什么**：检查初始化状态、读取/写入字段、预加载 skill 配置
- **给谁用**：所有 SKILL.md 启动时通过 `!` 语法调用
- **什么时候用**：skill 启动时预加载状态

以下脚本已删除（功能可被 agent 自身能力替代）：

| 文件 | 替代方式 |
|------|---------|
| xbase/scripts/include.py | agent 按需 Read references 文件；跨 skill 引用用路径直接读取；段提取由 agent 定位 markdown 段落 |
| xbase/scripts/artifact-create.py | agent 自行按 template 创建 |
| xcommit/scripts/git-context.py | agent 自行执行 git 命令 |
| xdecide/scripts/decision-log.py | agent 自行管理编号 |
| xtest/scripts/issues.py | agent 自行管理状态 |

### 项目文件

生成在用户项目中，由 x-skill 运行时创建和维护。skill 间的数据合同——读写双方遵循同一格式，修改格式需同步更新所有消费方。

**规范（\*-RULES.md）**

- **是什么**：项目特定的规范实例
- **怎么来的**：agent 在 guideline 指导下，将项目探测数据生成
- **包含什么**：具体的、可操作的规则条目
- **给谁用**：agent 执行 skill 时，作为判断和操作依据
- **什么时候用**：skill 运行时读取
- **不包含什么**：通用知识（guideline）

**记录（\*-LOG.md / \*-CHECKLIST.md / \*-ISSUES.md）**

- **是什么**：跨 session 持久化的过程数据
- **怎么来的**：按 template 创建空文件，按 standard 追加条目
- **包含什么**：具体的过程条目（Bug 记录、决策、测试结果、问题）
- **给谁用**：对应 skill 写入，跨 skill 读取（如 xDebug 读 TEST-ISSUES）
- **什么时候用**：skill 执行时写入和读取
- **不包含什么**：规范条目（那是规范类的事）

**状态枢纽（SKILL-STATE.md）**

- **是什么**：所有项目文件路径和初始化状态的索引
- **怎么来的**：xBase 初始化时创建和维护
- **包含什么**：每个 skill 的项目文件路径、初始化日期
- **给谁用**：所有 skill，启动时读取获取项目文件位置
- **什么时候用**：skill 启动时
- **不包含什么**：项目文件的内容本身

**工具（run.sh）**

- **是什么**：项目构建/启动/停止/日志读取命令的集合
- **怎么来的**：xBase 初始化时根据项目探测生成
- **包含什么**：可执行的 shell 命令
- **给谁用**：xDebug、xTest 执行时调用
- **什么时候用**：需要构建、启动或读取日志时
- **不包含什么**：业务逻辑

## 触发机制

Skill 通过三种方式激活：

| 方式 | 机制 | 示例 |
|------|------|------|
| 意图识别 | Claude 匹配用户消息与 skill 的 description，自动调用对应 skill | "这个按钮点了没反应" → xDebug |
| 固定锚点 | 嵌入在 skill 流程中的固定检查点，每次执行必经过 | xCommit 提交前必问审查、提交后必查决策 |
| 手动调用 | 用户输入 `/xname` 直接调用 | `/xreview`、`/xbase` |

Skill 间的条件衔接（如测试失败→xDebug、修复完成→xCommit）详见「Skill 间衔接矩阵」。

### 各 Skill 触发方式

| Skill | 意图识别（用户说…） | 固定锚点 |
|-------|-------------------|---------|
| xBase | — | — |
| xDebug | 报告 Bug、请求调试、排查问题 | — |
| xLog | 补日志、审查日志、修正日志 | — |
| xTest | 测试、验证功能、跑测试用例 | — |
| xReview | review 代码、审查变更、检查质量 | xCommit 提交前固定询问 |
| xCommit | 提交代码、保存变更 | — |
| xDoc | 维护文档、检查文档质量 | — |
| xDecide | 做决策、记录决策、回顾决策 | xCommit 提交后 AI 检测 |

所有 skill 均支持 `/xname` 手动调用。

## Skill 间衔接矩阵

### 三种衔接模式

| 模式 | 触发者 | 行为 | 适用条件 |
|------|--------|------|---------|
| 衔接调用 | 用户选择 | 跳转到目标 skill 的决策点入口，自动传递核心文件中的条目编号或路径 | 源和目标有直接因果关系（如测试失败→调试） |
| 提醒 | agent 展示 | 提示用户可能需要关注，不自动跳转 | 仅信息关联，无直接因果（如提交时可能遗漏文档） |
| 子 agent 调用 | agent 自动 | skill 内部调用另一个 skill 的子模式，无需用户确认 | 调用方完全掌握上下文，被调用方是轻量辅助模式（如 xDebug→xLog targeted） |

### 衔接关系

| 源 | 目标 | 触发条件 | 调用方式 |
|----------|-----------|---------|---------|
| xTest D5 | xDebug D1 | 用户选择"立即修复" | 衔接调用，传递 TEST-ISSUES.md 条目编号 |
| xDebug D4 | xLog D5 | 现有日志不足以定位问题 | 子 agent 调用，targeted 模式 |
| xDebug D7 | xCommit D1 | 修复完成，用户选择"提交变更" | 衔接调用 |
| xDebug D7 | xDecide D1 | 修复涉及技术决策 | 衔接调用，传递决策背景 |
| xDebug D7 | xTest D6 | 修复完成，需要回归验证 | 衔接调用，传递关联测试项 |
| xReview D4 | xDecide D1 | 发现设计级问题需记录决策 | 衔接调用，明确标注"超出 code review 范围" |
| xReview D5 | xCommit D1 | 审查通过，用户选择"提交变更" | 衔接调用 |
| xCommit D3 | xReview D1 | 提交前固定询问是否需要审查 | 衔接调用 |
| xCommit D4 | xDoc | 提交时检测到文档可能遗漏 | 提醒 |
| xCommit D5 | xDecide D1 | 提交成功后，AI 检测隐式决策 | 衔接调用 |
| xReview D4 | xDecide D1 | 代码变更违背已有决策 | 衔接调用，提示检查决策是否需要替代 |

### 跨 Skill 共享文件

衔接矩阵中的数据传递依赖以下共享文件（单一归属的规范/记录文件见各 skill 段）：

| 文件 | 写入者 | 读取者 | 传递的数据 |
|------|--------|--------|-----------|
| SKILL-STATE.md | xBase | 所有 skill | 核心文件路径、初始化状态 |
| run.sh | xBase | xDebug, xTest | 构建/启动/停止/日志读取命令 |
| TEST-ISSUES.md | xTest, xDebug | xTest, xDebug | 问题条目编号和状态（🔴→🟡→🟢→✅） |
| DECIDE-LOG.md | xDecide | xDecide, xDebug | 决策编号（D-NNN）和前提假设 |

## 典型工作流

衔接矩阵定义两两关系，以下是常见的端到端路径。括号表示可选步骤。

**Bug 修复**

1. xDebug 定位根因（→ xLog targeted 补日志）
2. 修复代码
3. xTest 回归验证（未通过则回步骤 1）
4. xCommit 提交（→ xReview 审查 → xDecide 决策捕获）

**测试驱动修复**

1. xTest 运行测试，发现失败
2. xDebug 定位修复（→ xLog targeted 补日志）
3. xTest 回归验证（未通过则回步骤 2）
4. xCommit 提交（→ xReview 审查 → xDecide 决策捕获）

**代码审查后提交**

1. xReview 审查变更，逐项修复
2. xCommit 提交（审查锚点跳过 → xDecide 决策捕获）

**日常提交**

1. xCommit（→ xReview 审查）→ 预检 → 提交（→ xDecide 决策捕获）

# xCommit

> 提交工作流：原子性保障 + 意图捕获 + 预检 + 规范化提交

**制品文件**

- **COMMIT_RULES.md**（规范）
  项目的提交规范——message 风格、预检脚本路径、暂存规则。
  - **创建**：xBase→xCommit 阶段 0，agent 在 guideline 指导下，从 git log 分析和 CLAUDE.md 提取生成
  - **读取**：xCommit 每次提交前自动加载，驱动预检和 message 生成
  - **更新时机**：项目提交规范变更时，通过 `reinit xcommit` 重新生成

**参考文件**

- **commit-rules-guideline.md**（生成指南）
  COMMIT_RULES.md 的生成指南——结构模板 + 质量原则，指导 agent 从项目数据产出高质量提交规范。
  - **消费场景**：xBase→xCommit 阶段 0 生成 COMMIT_RULES.md 时

### D1. 变更收集与原子性分析
收集工作区所有变更（staged + unstaged + untracked），分析变更内容，识别有几个独立的逻辑变更。
- **单一变更** → D2
- **多个变更** → D6（拆分提交）

### D2. 暂存管理
展示变更文件，引导选择暂存（不用 `git add -A`），检查无敏感文件/调试代码。
→ D3

### D3. 审查锚点
根据变更量判断是否需要提交前审查。
- **超过阈值** → 询问"提交前是否需要审查？"→ 衔接 xReview D1 或跳过 → D4
- **小变更**（typo、单行修复等）→ 自动跳过 → D4

### D4. 预检与文档完整性
运行 COMMIT_RULES.md 中的预检脚本，失败可自动修复（修复产生的新变更自动补充暂存）。推断变更类型，检查是否遗漏文档更新（建议不阻断）。
→ D5

### D5. 生成 message 与提交
AI 从 diff 提取 what，**主动追问用户 why**（变更动机/背景），按项目风格格式化。确认后执行 commit 并自动 push 到远程。提交成功后，AI 回顾对话和变更检测隐式决策：
- **有隐式决策** → 提醒衔接 xDecide D1 记录
- **无决策痕迹**（小变更）→ 自动跳过

### D6. 拆分提交
分析哪些文件/变更属于同一个逻辑变更，提出拆分方案（如"重构"和"新功能"分开）。审查锚点在首组触发一次。逐组走核心流程：暂存 → 预检 → message → 提交。决策捕获在末组触发一次。

# xLog

> 诊断日志治理：盲区补全 + 反模式纠正 + 冗余抑制

**核心文件**

- **LOG-RULES.md**（规范）
  项目的日志规范——可用的 Logger 列表、级别使用约定、消息格式要求。
  - **创建**：xBase→xLog 阶段 0，agent 在 guideline 指导下，从项目代码探测生成
  - **读取**：xLog 执行时以此为判断依据（扫描治理 D1-D4 + targeted D5）
  - **更新时机**：项目日志框架变更时，通过 `reinit xlog` 重新生成

**参考文件**

- **log-rules-guideline.md**（生成指南）
  LOG-RULES.md 的生成指南——结构模板 + 质量原则，指导 agent 从项目数据产出高质量日志规范。
  - **消费场景**：xBase→xLog 阶段 0 生成 LOG-RULES.md 时

**入口**

| 入口 | 触发方式 | 起始点 |
|------|----------|--------|
| 扫描治理 | 用户直接调用 | D1 |
| targeted | xDebug 衔接调用 | D5 |

### D1. 扫描与发现
自动收集近期变更文件，用户选择处理范围（最近变更 / 指定路径）。一次扫描两类问题：
- **盲区**：错误分支、guard 失败、状态转换、FFI 边界、异步回调入口
- **不规范**：print 替代 Logger、级别滥用、无上下文消息、Log and Throw、冗余日志、Logger 归属错误

**冗余日志抑制**：明确"不该加"的位置——纯计算函数、无副作用的 getter/setter、循环体内每次迭代、下游已有日志覆盖的调用点——抑制 AI 的"宁多勿少"倾向。
**消息三要素**：每条日志回答三个问题——谁/在哪（上下文）、为什么走到这（因果）、接下来查什么（可操作性）。
→ 展示发现摘要 → D2

### D2. 处理范围
用户选择处理哪些发现。
- **全部处理** → 执行补全 + 纠正 → D3
- **只补盲区** → 仅处理盲区类发现 → D3

### D3. 编译验证
补全/纠正后构建验证。
- **通过** → D4
- **失败**（≤3 次）→ 自动修复重试
- **失败**（超过 3 次）→ 用户选择回滚 / 手动修复 / 跳过

### D4. 迭代决策
汇报处理结果。
- **继续处理其他区域** → 回到 D1（新的范围选择），一次 session 覆盖多个区域
- **结束**（终点）

### D5. targeted 模式
被 xDebug 调用的轻量入口。针对特定路径 + 问题上下文快速补日志，不交互。编译验证：失败最多重试 2 次；仍失败则自动回滚所有日志变更并退出——子 agent 不阻塞主流程。

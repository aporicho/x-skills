# xReview

> 代码审查：粒度控制 + 影响面先行 + 多维度覆盖 + 置信度透明

**制品文件**

- **REVIEW_RULES.md**（规范）
  项目的代码审查规则——编码规范、命名约定、架构约束、禁止项。
  - **创建**：xBase→xReview 阶段 0，agent 在 guideline 指导下，从 CLAUDE.md 和代码扫描提取生成
  - **读取**：xReview 审查时逐条对照
  - **更新时机**：项目审查规范变更时，通过 `reinit xreview` 重新生成

**参考文件**

- **review-rules-guideline.md**（生成指南）
  REVIEW_RULES.md 的生成指南——结构模板 + 质量原则，指导 agent 从项目数据产出高质量审查规范。
  - **消费场景**：xBase→xReview 阶段 0 生成 REVIEW_RULES.md 时

### D1. 范围与粒度
用户选择审查目标（未提交变更 / 最近一次提交 / 指定路径或 git range）。计算变更量。
- **≤400 行** → 直接审查 → D2
- **>400 行** → 按模块或变更类型拆分，分批审查（SmartBear/Cisco 研究：超过此阈值检出率显著下降）→ 每批走 D2

### D2. 上下文加载与分析
查询 DECIDE_LOG.md 中的相关决策（避免建议违背已有决策）和 DEBUG_LOG.md 中的相关修复记录（关注已知易错点）。追踪变更的调用方、继承者、跨模块影响、风险评估。识别变更类型并调整七维度审查权重：
- **Bug 修复** → 重点正确性 + 影响面
- **新功能** → 重点可理解性 + 架构适配 + 测试伴随
- **重构** → 重点影响面 + 架构适配

→ D3

### D3. 七维度审查
正确性 / 影响面 / 可理解性 / 架构适配 / 健壮性 / 测试伴随 / 规范合规。每个发现标注 AI 置信度（高——规范/格式，机械性检查；中——边界/测试；低——逻辑/架构，提示人类重点复核）。分级反馈：🔴 Must-fix / 🟡 Should-fix / 🟢 Nit。发现超出 code review 范围的设计级问题时，明确标注并建议衔接 xDecide 做设计级讨论，而非试图在 review 中解决。
→ D4

### D4. 逐项处理
每个问题单独决策。
- **立即修复** → 执行修复
- **记录重构清单** → 记录后跳过
- **记录决策** → 衔接 xDecide D1
- **跳过**

全部处理完 → D5

### D5. 收尾与衔接
审查结束后汇总处理结果。
- **提交变更** → 衔接 xCommit D1
- **记录决策** → 衔接 xDecide D1
- **继续审查其他代码** → 回到 D1

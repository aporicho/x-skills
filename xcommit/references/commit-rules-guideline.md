# COMMIT-RULES.md 生成指南

> 供 agent 生成/审查 COMMIT-RULES.md 时参照，不是用户文档。
> 基于行业共识：Conventional Commits 1.0、Angular Commit Guidelines、Linux Kernel Patch Philosophy、Chris Beams "How to Write a Git Commit Message"。

---

## 结构模板

两层来源提取（不是凭空编）：
1. **CLAUDE.md**（如存在）→ 提取提交相关规则（禁止部分提交、文档完整性要求等）
2. **项目扫描**（始终执行）→ git log 风格分析 + 预检脚本探测 + 文档映射推导

```markdown
# COMMIT RULES

> 由 /xcommit 基于 git log 分析 + CLAUDE.md 生成

更新时间：YYYY-MM-DD
来源：[CLAUDE.md + 项目扫描 / 仅项目扫描]

## 提交消息风格

### 格式
| 项目 | 规范 | 来源 |
|------|------|------|
| 语言 | [如：中文] | [git log 分析] |
| 前缀 | [如：无前缀 / feat: fix: 等] | [git log 分析] |
| 长度 | [如：简短一行] | [git log 分析] |
| 示例 | [从最近提交提取的典型示例] | [git log 分析] |

## 暂存规则

| 规则 | 检查点 | 来源 |
|------|--------|------|
| [如：禁止 git add -A] | [检查暂存命令] | [CLAUDE.md / 项目扫描] |
| [如：每次提交包含所有变更] | [git status 确认无遗漏] | [CLAUDE.md / 项目扫描] |

## 预检脚本

| 脚本 | 用途 | 阻断级别 |
|------|------|----------|
| [如：scripts/preflight.py] | [预提交检查] | [阻断 / 警告] |

## 文档完整性检查

### 变更类型 → 文档映射
| 变更类型 | 判断依据 | 应更新的文档 | 来源 |
|----------|----------|-------------|------|
| Bug 修复 | diff 含修复逻辑 | DEBUG-LOG.md | [CLAUDE.md / 项目扫描] |
| Bug 修复（来自 TEST-ISSUES） | diff 含修复逻辑 + TEST-ISSUES 条目 | DEBUG-LOG.md + TEST-ISSUES.md 状态更新 | [项目扫描] |
| 架构/决策 | 新增/删除模块 | DECIDE-LOG.md | [CLAUDE.md / 项目扫描] |
| 新功能 | 新增文件/函数 | [对应功能文档] | [项目扫描] |

## 提交禁忌（汇总速查）

> 从上面各节提取的禁忌项汇总。

- [ ] [禁忌 1]
- [ ] [禁忌 2]
```

## 更新时机

- `reinit` 时重新生成
- 项目提交规范变化时
- CLAUDE.md 提交相关规则更新后
- 用户手动调整后以用户版本为准

## 质量原则

### A. 消息风格（Message Style）

#### 好消息的两层信息

1. **What**（主题行）— 这次提交做了什么，一句话说清
2. **Why**（body）— 为什么这样做，代码本身说明 How，消息说明 Why

#### 主题行质量标准

| 标准 | 判断依据 |
|------|---------|
| 长度有上限 | 指定具体字符数（50-72 之间取一个值），是软限制还是硬断 |
| 语言明确 | 中文或英文，不是"保持一致" |
| 语气一致 | 祈使式或陈述式，选定一种 |
| 标点明确 | 末尾加不加句号、首字母是否大写 |
| 可扫视 | 前缀/类型（如有）让人不读全文就能判断提交性质 |

#### 正反例对比

| 原则 | 坏例子 | 好例子 |
|------|--------|--------|
| 说明意图而非文件 | `"Update file.swift"` | `"修复拖拽时元素位置偏移"` |
| 不用 and 连接 | `"修复 Bug 并重构状态管理"` | 应拆成两个提交 |
| 不用单词敷衍 | `"Bugfix"` | `"修复 guard 失败时未回退状态导致 UI 卡死"` |
| body 解释 why | 只有主题行 | body 补充：`"旧逻辑假设回调必定触发，但在 App 进后台时回调被取消"` |

### B. 工作流一致性（Workflow Alignment）

#### 必须声明的策略

COMMIT-RULES.md 必须明确项目采用的工作流策略，因为不同策略对规范的要求截然不同：

| 策略 | 对消息格式的影响 | 对预检的影响 |
|------|----------------|-------------|
| 全程原子提交 | 每个提交都必须合规 | 每个提交都必须通过预检 |
| 分支随意 + 合并前 squash | 只有最终 squash 提交需要合规 | 预检只在合并时运行 |

#### 类型列表边界

如果项目使用类型前缀：
- 类型数量 ≤ 10，每种配一个项目实际的示例消息
- 每种类型的边界清晰，不存在"这个该标 feat 还是 chore"的歧义
- 无类型前缀的项目不要硬套 Conventional Commits

#### 规范与工具链匹配

- 如果项目接入了 changelog 自动生成 / SemVer 自动化 → 消息格式必须匹配工具要求
- 如果不接入任何自动化工具 → 不需要机器可读格式，人类可读即可
- commitlint 配置（如有）与文档描述的规则必须一致，不能一个松一个紧

### C. 预检脚本（Pre-check Scripts）

#### 好的预检配置的三个属性

| 属性 | 判断标准 |
|------|---------|
| 快速 | 秒级完成，不拖慢开发节奏 |
| 可定位 | 报错指向具体文件和行号，不是笼统的"检查失败" |
| 阻断级别明确 | 每个脚本标明"阻断"（必须通过）或"警告"（提示但不阻止） |

#### 阻断 vs 警告的决策框架

| 应当阻断 | 应当警告 |
|---------|---------|
| 编译失败 | 代码复杂度超标 |
| 机密信息泄露 | TODO/FIXME 数量增加 |
| 消息格式不合规（如有 commitlint） | 文档未同步更新 |

#### 脚本条目完整性

每个预检脚本条目必须包含：脚本路径、用途说明、运行命令、阻断级别。缺任何一项 = 不可执行。

### D. 文档完整性映射（Documentation Mapping）

#### 映射规则的质量标准

| 标准 | 判断依据 |
|------|---------|
| 指向具体文件 | `"更新 DEBUG-LOG.md"` 而非 `"更新相关文档"` |
| 基于项目实际 | 映射关系中的文档文件必须在项目中实际存在 |
| 判断依据可操作 | `"diff 含修复逻辑"` 而非 `"如果是 Bug 修复"` |
| 不阻断提交 | 文档提醒是建议，不是硬性阻断（避免过度摩擦导致 --no-verify） |

### E. 生成规范时的自检清单（Self-Check for Generation）

生成或更新 COMMIT-RULES.md 后，逐项核对：

- [ ] **消息风格有具体数字** — 主题行长度上限是确切数字，不是"简短"
- [ ] **语言/语气/标点规则明确** — 不是"保持一致"，是"用中文/祈使式/末尾不加句号"
- [ ] **工作流策略已声明** — 全程原子还是 squash，不能省略
- [ ] **类型列表（如有）≤ 10 种** — 每种有项目实际的示例消息
- [ ] **预检脚本来自实际探测** — 路径、命令、阻断级别都有，不是猜的
- [ ] **文档映射指向项目实际存在的文件** — 逐个确认文件存在
- [ ] **规范与工具链匹配** — commitlint 配置（如有）与文档一致
- [ ] **禁忌汇总与 CLAUDE.md 一致** — 逐条对照
- [ ] **示例来自项目实际 git log** — 不是通用模板

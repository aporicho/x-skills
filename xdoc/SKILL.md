### 参数处理（`$ARGUMENTS`）

- **空** → 正常走「选择操作」询问
- **`巡检`** → 跳过「选择操作」，直接进入「巡检」
- **`同步`** → 跳过「选择操作」，直接进入「同步」
- **`重组`** → 跳过「选择操作」，直接进入「重组」
- **`瘦身`** → 跳过「选择操作」，直接进入「瘦身」
- **其他文本** → 作为指定文件/目录路径，对该路径执行巡检（路径不存在 → 输出"路径不存在：{path}"，停止）

| `DOC-RULES.md` | 文档规范（目录结构 + 检查脚本 + 映射规则） | `references/doc-rules-guideline.md` |

### 初始化检查

查看上方预加载输出：
- 含 `not_found` → 输出"xdoc 尚未初始化，请先运行 `/xbase`"，**立即停止，不执行任何后续步骤，不尝试替代方案**
- 含 `initialized` → 从输出中读取 `## xdoc` 段的 `doc_rules` 字段，得到 DOC-RULES.md 路径
  - 文件存在 → 进入「选择操作」
  - 文件不存在 → 输出"DOC-RULES.md 不存在（路径：{path}），请运行 `/xbase reinit xdoc` 重新生成"，**立即停止，不执行任何后续步骤，不尝试替代方案**

> **为什么必须停止**：xdoc 的所有操作（巡检/同步/重组/瘦身）依赖 DOC-RULES.md 中的目录结构、检查脚本和映射规则。没有这个文件，agent 只能凭猜测工作，产出的结果既不完整也不可靠。与其给用户一个低质量的结果，不如明确告知缺少前提，让用户先完成初始化。

### 选择操作

用 AskUserQuestion：

```
问题：文档维护。选择操作：
选项：
- 巡检（断链 / 格式 / 结构 / 新鲜度 / 覆盖率）
- 同步（代码变更 → 文档更新）
- 重组（目录结构 → 约定对齐）
- 瘦身（删冗余 / 合并过时）
- Other → 输入具体需求
```

### 巡检

读取 DOC-RULES.md（使用初始化检查中获取的路径）。

**第一步：运行检查脚本**（从「检查脚本」列表逐个运行，捕获输出，记录每个脚本覆盖了哪些维度）

**第二步：用内置逻辑补全脚本未覆盖的维度**（已被脚本覆盖的维度跳过）

五个维度：

1. **断链检测**：扫描 DOC-RULES.md「文档目录结构」中所有目录的 markdown 文件，验证内部链接目标文件存在、外部链接可访问
2. **格式合规**（按 DOC-RULES.md「格式规范」执行）：标题层级、代码块语言标注、空文件或占位文件
3. **结构合规**：对比实际目录结构 vs DOC-RULES.md「文档目录结构」中的约定，列出偏差
4. **新鲜度评估**：运行 `git log --oneline -10`，对照 DOC-RULES.md「代码-文档映射」，识别有代码变更但文档未同步的情况
5. **覆盖率分析**：读取 DOC-RULES.md「代码-文档映射」中列出的代码目录，扫描其中的公开接口和模块，列出无对应文档的模块

按严重程度排序（断链 > 格式 > 结构 > 新鲜度 > 覆盖率），输出健康报告（终点）。报告格式：

```
# 文档健康报告

> 检查时间：YYYY-MM-DD | 范围：[全局 / 指定路径]

## 健康摘要

| 维度 | 状态 | 问题数 |
|------|------|-------|
| 断链检测 | [通过/有问题] | N |
| ...  | ...  | ... |

## 1. 断链检测
[具体发现，每个问题包含文件路径和描述]

## 2. 格式合规
...

## 问题汇总（按严重程度排序）

| 编号 | 维度 | 严重程度 | 问题描述 | 文件 |
|------|------|---------|---------|------|
| 1 | [维度] | [高/中/低] | [描述] | [路径] |
```

### 同步

1. **获取最近代码变更**：运行 `git log --oneline -20` 和 `git diff HEAD~10..HEAD --stat`（提交数不足 10 则用实际提交数），展示给用户确认范围：

```
问题：同步范围（以上最近 N 次提交）。是否调整？
选项：
- 使用以上范围
- Other → 输入 commit hash 或提交数量
```

2. **定位需要更新的文档**：读取 DOC-RULES.md「代码-文档映射」，对照变更文件识别应更新的文档。
   - 没有匹配项 → 输出"未发现需要同步的文档"，停止

3. **逐个确认漂移**：对每个需要更新的文档：
   - **Read 打开该文档**，通读当前内容
   - **Read 打开对应的变更文件**（如源码、配置），对比具体差异
   - 不要仅凭 `git diff --stat` 的文件名猜测漂移，要读文件内容确认实际偏差

4. **草拟并逐项确认**：对每个确认存在漂移的文档，**直接草拟具体的更新内容**（不只是提醒），用 AskUserQuestion 逐项展示：

```
问题：[文档路径]
  代码变更：[相关提交或文件]
  草拟更新：
  [具体的文档更新内容]
选项：
- 写入
- Other → 输入修改后的内容再写入
- 跳过
```

5. 选择"写入"或输入修改内容 → 执行写入
→「验证与汇报」

### 重组

1. **扫描当前结构**：用 Glob 列出 DOC-RULES.md「文档目录结构」中定义的各目录的实际文件树

2. **对比约定结构**：读取 DOC-RULES.md「文档目录结构」，列出差异（多余目录、缺失目录、位置不对的文件）。
   - 没有差异 → 输出"结构已符合约定"，停止

3. **用 AskUserQuestion 展示重组建议**（逐项，不一次全列）：

```
问题：[序号/总数] 建议重组：
  操作：移动 / 合并 / 拆分
  详情：[文件] → [目标路径]
  影响：涉及 N 处引用
选项：
- 执行
- 跳过
- 取消全部
```

4. 每项确认执行后：
   - 移动/合并/拆分文件
   - **自动更新所有引用**：Grep 搜索所有引用旧路径的文件，Edit 替换为新路径
   - **重建索引**：检查 DOC-RULES.md「检查脚本」中是否有索引生成脚本（如 `generate_index`），有则运行；无则跳过

→「验证与汇报」

### 瘦身

1. **读取忽略列表**：检查 DOC-RULES.md 是否有 `## 瘦身忽略列表` 段，有则读取，后续跳过列表中的文件

2. **冗余检测**：扫描 DOC-RULES.md「文档目录结构」中所有目录的 markdown 文件，提取各文件的标题列表（H1-H3），识别跨文件标题高度相似（描述同一概念）但内容不一致的情况

3. **过时检测**：
   - **无引用文档**：对每个文档文件名，Grep 搜索项目中是否有其他文件引用它，无引用 → 候选
   - **已废弃内容**：提取文档中提到的模块名、文件路径、类名等，Grep 代码库确认是否仍存在，已不存在 → 候选

4. 冗余和过时检测均无发现 → 输出"未发现冗余或过时内容"，停止

5. **逐项用 AskUserQuestion 展示建议**：

```
问题：[检测类型] [文件路径]
  问题：[描述]
  建议：[删除 / 合并到 {目标文件}]
选项：
- 执行
- 跳过
- 忽略（写入 DOC-RULES.md 忽略列表，下次跳过）
```

选择"忽略"时：在 DOC-RULES.md `## 瘦身忽略列表` 段追加该文件路径（段不存在则创建）。

→「验证与汇报」

### 验证与汇报

统计本次操作修改的文件数量（N）。

N > 10 时：运行 DOC-RULES.md「批量编辑验证」中指定的脚本确认修复质量。

用 AskUserQuestion 汇报：

```
问题：文档维护完成。
  执行了 X 项操作，修改 N 个文件（跳过 Y 项）。
  下一步？
选项：
- 重新巡检（确认修复效果）
- 提交变更（→ /xcommit）
- 结束
```

## 关键原则

- **规则从文件读取** — 文档目录、检查脚本、映射规则基于 DOC-RULES.md，`/xbase reinit xdoc` 时重新生成
- **脚本优先** — DOC-RULES.md 中有检查脚本就用，没有才用内置基础检查
- **同步要草拟，不只是提醒** — 给出具体更新内容，不只列出"需要更新"
- **重组逐项确认** — 每个重组操作单独展示影响范围，不批量执行
- **重组要更新引用** — 移动文件后自动搜索并替换所有引用路径
- **巡检只读** — 巡检不做任何修改，只输出报告
- **忽略要持久化** — 写入 DOC-RULES.md 忽略列表，不是一次性跳过
- **批量修复后验证** — 10+ 文件修改后按 DOC-RULES.md 运行验证脚本
- **选项优先于打字** — Other 兜底自由输入
- **每轮只问一个问题** — 不堆叠

### 阶段 1：确定范围

用 AskUserQuestion：

```
问题：审查什么代码？
选项：
- 未提交变更（git diff）
- 最近一次提交（git show HEAD）
- 指定路径（→ Other 输入路径或 git range）
- Other → 自由输入
```

### 阶段 2：执行审查

#### 步骤 1：获取变更

- 未提交变更 → `git diff` + `git diff --cached`
- 最近提交 → `git show HEAD`
- 指定路径 → 读取文件
- git range → `git diff <range>`

识别变更类型（Bug 修复 / 新功能 / 重构 / 混合），用于步骤 3 调整审查权重。

#### 步骤 1.5：历史知识加载

从 SKILL-STATE.md 获取路径，用变更涉及的模块/功能关键词搜索：
- **DECIDE-LOG.md**：相关决策条目，确保审查建议不违背已有决策
- **DEBUG-LOG.md**：相关修复记录，关注已知易错点

#### 步骤 2：影响面分析（先于代码审查执行）

> 核心问题：**改了这里，别处会不会炸？**
>
> 来源：Change Impact Analysis（Yau et al. 1980 "ripple effect"）、Google eng-practices "Context"。

对每个变更的函数/类型/接口，追踪影响链：

1. 用 Grep 搜索所有调用方（函数名、类型名、协议/接口名）
2. 用 Grep 搜索所有继承者/实现者（子类、协议遵从）
3. 检查变更是否涉及公共 API / 跨模块接口 / FFI 边界
4. 检查变更的参数签名、返回值、行为语义是否改变

产出**影响面摘要**：

```
影响面：
- 直接调用方：N 处（列出文件:行号）
- 接口/协议变更：是/否
- 跨模块影响：是/否（哪些模块）
- 风险评估：高/中/低
```

影响面为"高"时，在后续审查中提高对应检查的优先级。

#### 步骤 3：七维度审查

读取 REVIEW-RULES.md（路径从 SKILL-STATE.md 的 `review_rules` 字段获取），按以下维度审查。根据变更类型调整权重：

- **Bug 修复** → 重点 A 正确性 + B 影响面
- **新功能** → 重点 C 可理解性 + D 架构适配 + F 测试伴随
- **重构** → 重点 B 影响面 + D 架构适配

**A. 正确性（Correctness）**

> 核心问题：**代码做对了吗？**

- 逻辑是否正确实现了预期行为（对照 commit message / PR 描述 / 用户说明）
- 边界条件：空值、零值、越界、最大最小、空集合
- 条件判断：是否取反、遗漏分支、短路求值陷阱
- 异步时序：回调顺序、竞态条件、取消处理
- 数据流：变量在使用前是否已正确赋值，类型转换是否安全

**B. 影响面（Impact）**

> 核心问题：**改了这里，别处会不会炸？**

基于步骤 2 的影响面摘要：
- 调用方是否需要同步修改
- 接口变更是否向后兼容
- 状态变更是否有意外的副作用传播
- 删除/重命名是否有遗漏的引用

**C. 可理解性（Clarity）**

> 核心问题：**六个月后的人能看懂吗？**
>
> 来源：Google "Complexity" 维度、CLAUDE.md "避免 over-engineering"。

- 命名：函数名、变量名是否传达意图（`calculateGrossRevenue` vs `calcRev`）
- 函数职责：一个函数是否只做一件事，是否过长（超过屏幕高度）
- 过度工程化：是否为不存在的未来需求提前抽象、是否引入不必要的泛型/协议/中间层
- 注释：复杂逻辑是否有注释解释"为什么"（不是"是什么"）
- 死代码：注释掉的代码、未使用的变量/导入

**D. 架构适配（Design Fit）**

> 核心问题：**放对地方了吗？符合系统设计吗？**

逐条核对 REVIEW-RULES.md `## D. 架构适配` 中的项目特有规则：
- 分层依赖：是否违反依赖方向（如内层依赖外层）
- 模块职责：新增代码是否放在正确的模块/目录
- 设计模式：是否遵循项目现有模式（如命令模式、注册模式）
- 重复代码：是否存在可合并的重复逻辑

**E. 健壮性（Robustness）**

> 核心问题：**异常情况下会不会崩？**

- 错误处理：catch/guard/Result 是否覆盖失败路径，错误信息是否有上下文
- 资源管理：内存泄漏、文件句柄未关闭、循环引用
- 并发安全：共享状态保护、主线程操作
- 安全漏洞：注入、XSS、硬编码密钥、权限检查
- 逐条核对 REVIEW-RULES.md `## E. 健壮性` 中的项目特有检查点

**F. 测试伴随（Test Coverage）**

> 核心问题：**变更有对应测试吗？**
>
> 来源：Google "Ask for tests as appropriate for the change"。

- 新增功能是否有对应的单元测试 / 集成测试
- Bug 修复是否有回归测试（防止问题复现）
- 测试是否覆盖正向和反向路径
- 测试命名是否描述预期行为（`test_failsWhenDuplicateEmail` vs `test_email`）
- 无测试时：是否属于合理豁免（纯 UI、配置变更等）

**G. 规范合规（Standards）**

> 核心问题：**符合项目规矩吗？**

逐条核对 REVIEW-RULES.md `## G. 规范合规` 中的规则：
- 项目禁忌：是否违反任何禁止规则
- 项目必须：是否遵守所有必须规则
- 编码规范：命名风格、缩进、注释语言
- 文档同步：代码变更是否需要同步更新文档（如新增 API 要更新文档、修改行为要更新 README）

#### 步骤 4：汇总发现，分级反馈

将所有发现按严重程度分级：

| 级别 | 含义 | 处理 |
|------|------|------|
| 🔴 Must-fix | 阻塞合并：正确性错误、安全漏洞、数据丢失风险 | 必须修复 |
| 🟡 Should-fix | 应当修复：架构偏差、缺少测试、可理解性差 | 强烈建议修复 |
| 🟢 Nit | 可选改进：命名微调、注释补充、风格细节 | 不阻塞，有空再改 |

展示审查摘要，用 AskUserQuestion：

```
问题：审查完成。[影响面摘要一句话]
  🔴 Must-fix：X 项
  🟡 Should-fix：Y 项
  🟢 Nit：Z 项
  [列出每项的维度、位置、问题、建议]
选项：
- 逐项处理（→ 逐个决策）
- 只处理 Must-fix
- 全部处理
- 看详细
- 结束
```

#### 步骤 5：逐项决策

对每个问题（按严重程度从高到低），用 AskUserQuestion：

```
问题：[🔴/🟡/🟢] [维度] [path:line]
  问题：[描述]
  建议：[修复方案]
选项：
- 立即修复
- 记录到重构清单
- 记录决策（→ /xdecide）
- 跳过
```

选择"立即修复"时：执行修复 → 读取修复后代码确认 → 继续下一项。

### 阶段 3：收尾

汇总处理结果，用 AskUserQuestion：

```
问题：审查结束。共 N 项发现（修复 X / 记录 Y / 跳过 Z）。下一步？
选项：
- 提交变更（→ /xcommit）
- 记录决策（→ /xdecide）
- 继续审查其他代码（→ 回阶段 1）
- 结束
```

## 关键原则

- **改善优于完美** — 代码只要确实提升了系统整体健康度就应接受，不追求完美（Google 黄金标准）
- **影响面先行** — 先追踪变更的调用方和依赖链，带着影响面认知审查代码
- **七维度覆盖** — 正确性、影响面、可理解性、架构适配、健壮性、测试伴随、规范合规
- **严重程度分级** — Must-fix / Should-fix / Nit，帮用户分清轻重缓急
- **逐项决策** — 每个问题单独决策，不堆叠
- **修复即时验证** — 修复后读取代码确认，不盲信
- **上下文感知** — 根据变更类型（Bug修复/新功能/重构）调整审查权重
- **规则从项目提取** — 基于 CLAUDE.md + 代码扫描，不硬编码

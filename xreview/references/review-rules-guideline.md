# REVIEW-RULES.md 生成指南

> 供 agent 生成/审查 REVIEW-RULES.md 时参照，不是用户文档。
> 基于行业共识：Google eng-practices、Google SWE Book Ch.9、Microsoft Engineering Fundamentals、Yelp Code Review Guidelines。

---

## 结构模板

两层来源提取（不是凭空编）：
1. **CLAUDE.md**（如存在）→ 关键词搜索提取禁忌/必须/规范/架构规则
2. **代码扫描**（始终执行）→ 推导命名风格、缩进、目录结构、架构分层、错误处理模式、测试框架等

```markdown
# CODE REVIEW RULES

> 由 /xreview 基于代码扫描 + CLAUDE.md 生成

更新时间：YYYY-MM-DD
来源：[CLAUDE.md + 代码扫描 / 仅代码扫描]

## D. 架构适配

> 项目特有的架构规则。审查维度 D 逐条核对此节。

### 分层与依赖
| 层级 | 职责 | 依赖方向 | 来源 |
|------|------|----------|------|
| [如：core/] | [领域逻辑] | [不依赖外层] | [CLAUDE.md / 目录结构推导] |

### 模块职责
| 模块/目录 | 职责边界 | 来源 |
|-----------|----------|------|
| [如：Bridge/] | [FFI 桥接，跨语言调用] | [代码结构推导] |

### 设计模式
| 模式 | 用在哪里 | 检查点 | 来源 |
|------|---------|--------|------|
| [如：命令模式] | [Undo/Redo] | [新操作是否通过命令系统] | [CLAUDE.md / 代码推导] |

## E. 健壮性

> 项目特有的健壮性检查点。审查维度 E 逐条核对此节。

### 项目特有检查点
| 检查点 | 说明 | 来源 |
|--------|------|------|
| [如：FFI 内存安全] | [跨语言指针/字符串的生命周期管理] | [代码模式推导] |
| [如：主线程 UI 操作] | [UI 更新必须在主线程] | [代码模式推导] |

## G. 规范合规

> 项目特有的编码规范。审查维度 G 逐条核对此节。

### 禁忌（违反则 Must-fix）
| 规则 | 检查点 | 来源 |
|------|--------|------|
| [如：禁止 print()] | [检查是否存在 print() 调用] | [CLAUDE.md / 代码推导] |

### 必须（遗漏则 Should-fix）
| 规则 | 检查点 | 来源 |
|------|--------|------|
| [如：日志用 Log.xxx] | [检查日志调用是否使用项目日志框架] | [CLAUDE.md / 代码推导] |

### 编码规范
| 项目 | 规范 | 来源 |
|------|------|------|
| 缩进风格 | [如：4 空格] | [代码扫描] |
| 命名风格 | [如：Swift camelCase / Rust snake_case] | [代码扫描] |
| 注释语言 | [如：中文] | [代码扫描 / CLAUDE.md] |
| 文件组织 | [如：MARK 分段] | [代码扫描] |

### 文档同步规则
| 变更类型 | 应同步更新 | 来源 |
|---------|-----------|------|
| [如：新增公共 API] | [更新 API 文档] | [CLAUDE.md / 项目惯例] |
| [如：修复 Bug] | [更新 DEBUG-LOG.md] | [CLAUDE.md] |

## 项目禁忌（汇总速查）

- [ ] [禁忌 2]
```

## 与 SKILL.md 审查维度的对应关系

SKILL.md 定义了七个审查维度（A-G），其中 A/B/C/F 的检查标准是通用的（写在 SKILL.md 中），D/E/G 的检查标准是项目特有的（写在本文件中）：

| 维度 | 标准来源 |
|------|---------|
| A. 正确性 | 通用（SKILL.md 内置） |
| B. 影响面 | 通用（SKILL.md 内置） |
| C. 可理解性 | 通用（SKILL.md 内置） |
| D. 架构适配 | **项目特有（REVIEW-RULES.md `## D`）** |
| E. 健壮性 | 通用 + **项目特有（REVIEW-RULES.md `## E`）** |
| F. 测试伴随 | 通用（SKILL.md 内置） |
| G. 规范合规 | **项目特有（REVIEW-RULES.md `## G`）** |

## 更新时机

- `reinit` 时重新生成
- 项目架构发生变化时
- CLAUDE.md 规则更新后
- 用户手动调整后以用户版本为准

## 质量原则

### A. 规则质量（Rule Quality）

#### 每条规则的四个必要属性

| 属性 | 判断标准 | 反例 |
|------|---------|------|
| 可操作 | 给出明确的 pass/fail 条件，不是模糊表述 | `"架构要合理"` → 应改为 `"禁止 core/ 层调用 infra/ 层"` |
| 可验证 | 两个独立审查者读同一条规则，对同一段代码给出相同结论 | `"代码要易读"` 无法独立验证 |
| 范围精确 | 一条规则覆盖一个问题，不混合多个关注点 | `"函数不能太长且命名要好"` 应拆成两条 |
| 有理由 | 能回答"为什么这样要求"，基于工程原则而非个人偏好 | `"禁止用三元运算符"` 缺少理由 |

#### 什么应该放进 REVIEW-RULES.md

- 项目特有的架构约束（哪层不能依赖哪层、哪些模块的职责边界）
- 特定技术栈的危险操作（FFI 内存安全、主线程 UI 操作、跨语言生命周期管理）
- 项目级禁忌（从 CLAUDE.md 提取的硬性禁止项）
- 必须同步更新的文档列表（DEBUG-LOG、DECIDE-LOG 等）

#### 什么不应该放进 REVIEW-RULES.md

- 能被 linter/formatter 自动检测的格式规范（交给工具，人工审查不重复）
- 通用编程原则（SOLID、DRY 等，不需要在项目规范中重复）
- 主观偏好（括号位置、空行数量等，应由格式化工具统一）

### B. 结构完整性（Structural Completeness）

#### 三个项目特有维度的检查要点

**D. 架构适配**
- 分层依赖关系是否完整覆盖所有层（不能只列两层漏掉第三层）
- 模块职责表是否覆盖项目中所有主要目录/模块
- 设计模式列表是否反映代码中实际使用的模式（而非通用推荐）

**E. 健壮性**
- 检查点是否针对项目技术栈的实际风险点（FFI 项目有内存安全、UI 项目有主线程约束）
- 是否区分了"程序能继续"和"程序不能继续"两类故障
- 每个检查点是否有明确的检查方法（在代码中找什么）

**G. 规范合规**
- 禁忌项是否与 CLAUDE.md 完全一致（一条不漏、一条不多编）
- 编码规范是否从项目实际代码扫描得出（不是语言通用规范）
- 文档同步规则是否指向项目实际存在的文档文件

#### 来源标注

每条规则必须标注来源：`[CLAUDE.md]`、`[代码扫描]`、`[目录结构推导]`。无来源的规则 = 凭空编的规则，必须删除或补充来源。

### C. 一致性（Consistency）

#### 规则间无矛盾

- 不同维度（D/E/G）的规则之间不存在冲突场景
- 当两条规则可能冲突时（如"架构严格分层" vs "性能优先"），必须说明优先级或给出决策指引
- 禁忌汇总表与各节内容一一对应，不多不少

#### 与其他核心文件无重复

- REVIEW-RULES.md 中的文档同步规则 ⊆ COMMIT-RULES.md 中的文档映射（不矛盾、不重复）
- REVIEW-RULES.md 中的禁忌项 = CLAUDE.md 中的对应子集（不增不减）

### D. 生成规范时的自检清单（Self-Check for Generation）

生成或更新 REVIEW-RULES.md 后，逐项核对：

- [ ] **每条规则有明确的 pass/fail 标准** — 不是"应该""建议"等模糊表述
- [ ] **每条规则标注了来源** — CLAUDE.md / 代码扫描 / 目录结构推导
- [ ] **D 节覆盖所有分层和主要模块** — 对照项目实际目录结构验证
- [ ] **E 节检查点针对项目技术栈** — FFI 项目有内存安全、UI 项目有主线程约束、异步项目有并发安全
- [ ] **G 节禁忌与 CLAUDE.md 完全一致** — 逐条对照，无遗漏无多编
- [ ] **G 节编码规范从代码扫描得出** — 缩进、命名、注释语言等与项目实际一致
- [ ] **排除了 linter 可自动检测的内容** — 人工审查规则不重复工具能力
- [ ] **规则间无矛盾** — 特别检查跨维度的潜在冲突
- [ ] **禁忌汇总表与各节一一对应** — 不多不少

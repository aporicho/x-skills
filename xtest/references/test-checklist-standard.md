# TEST_CHECKLIST.md 书写标准

## 创建位置

存放在 SKILL-STATE.md 的 `doc_dir` 目录下。

## 编号规则

从项目代码结构动态推导 ID 前缀和编号体系，不硬编码。

## 分类判断标准

| 类别 | 判断条件 | 典型场景 |
|------|---------|---------|
| 🤖 自动化 | 结果确定性强（相同输入 = 相同输出）、执行频率高、无主观判断 | 单元测试、API 返回值、数据持久化、边界条件 |
| 👤 手动 | 涉及视觉/交互体验、需要主观判断、探索性验证 | 动画流畅度、拖拽手感、布局协调性、可访问性 |
| 🤝 结合 | 自动化执行核心逻辑 + 人工验证体验质量 | 自动触发操作后人工检查渲染结果、自动准备数据后人工探索边界 |

简明决策：能用代码断言 pass/fail 的 → 🤖；需要人眼/人手判断的 → 👤；两者各占一半的 → 🤝。

## 测试项质量标准

### 手动测试项必备要素

每条手动测试项应该能让**任何人独立执行并判断通过/失败**，不需要额外解释。"测试项"列的内容应包含：

- **操作**：具体的 1-2-3 步骤（不是"测试拖拽功能"，是"将节点 A 拖拽到容器 B 上方"）
- **预期结果**：可观察的具体现象（不是"正常工作"，是"节点吸附到容器内，容器边框高亮消失"）

### 粒度判断

| 信号 | 含义 |
|------|------|
| 测试项失败时，30 秒内能知道哪里出问题 | 粒度合适 |
| 测试项失败可能有多种原因，需要排查 | 太粗，应拆分 |
| 一个功能改动导致 5+ 个测试项需要更新 | 太细或重叠，应合并 |
| 能用一句话说清"谁在什么条件下做什么，期望看到什么" | 粒度合适 |

## 迁移旧格式

如果检测到已有测试清单但格式不同：
1. 用 AskUserQuestion 展示差异，询问是否迁移
2. 用户同意 → 保留所有原始测试结果（通过/失败/待测状态），套用新格式
3. 用户拒绝 → 保持原样，后续更新时仍遵循原格式
